# 架電KPI分析フォーマット

## 目的
架電結果を単なる数値報告で終わらせず、「どこで落ちているか」「何を直すか」まで言語化する。

## 基本KPI
```text
・電話数：{n}件
・着電率：{x}%
・受付拒否数：{n}件（受付拒否率：{x}%）
・受付資料請求数：{n}件（受付資料請求率：{x}%）
・対象者通話：{n}件（対象者通話率：{x}%）
・対象者拒否数：{n}件（対象者拒否率：{x}%）
・本人資料請求数：{n}件（本人資料請求率：{x}%）
・アポイント数：{n}件（アポイント率：{x}%）
・不通数：{n}件（不通率：{x}%）
```

## 報告フォーマット
```md
## 定性的な報告
- {今回の架電全体の状態}
- {どの指標が良く、どこで落ちているか}
- {ログ上の特徴}

## 乖離している場合
### 原因
- {リスト/受付突破/本人接続後/トーク/タイミング/商材理解のどこが原因か}

### 改善
- {トーク・リスト・運用・商談接続で何を直すか}

### 対策
- {次回までの具体アクション}
```

## ボトルネック判定
| 症状 | 主因仮説 | 修正方向 |
|---|---|---|
| 着電率が低い | 時間帯・リスト鮮度 | 時間帯変更・再架電設計 |
| 受付拒否率が高い | 冒頭で営業判定 | 商材説明を消し、担当確認型へ |
| 対象者通話率が低い | 受付突破ワード不一致 | 部署/担当名/確認事項を明確化 |
| 対象者拒否率が高い | 本人接続後30秒の説明過多 | 課題切り分け質問へ変更 |
| 資料請求が多いがアポ少 | 温度中で止まる | 追撃フォロー台本・次回連絡許可 |
| アポは多いが受注しない | 課題未確定アポ | 課題確定済アポ定義へ変更 |

## 原則
- 数値だけで評価しない。
- 受付前/受付後/本人接続後のどこで落ちているかを見る。
- 本人拒否ログは必ず分類する。
- 台本修正はログの言葉から行う。
