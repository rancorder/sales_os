# 2次商談後フォロー・IS比較検証OS

更新日：2026-06-29  
抽出元：FirstFan / AGERUケア案件、ダッドウェイ様・ほしゆう様の2次商談後フォロー設計、SP＋IS比較検証の議論

---

## 0. 位置付け

このOSは、営業支援OSの中でも「2次商談後に案件が止まりやすい局面」と「通常架電だけでは勝ち筋が見えにくい局面」を扱う補助エンジンである。

既存の `patterns/sales_support_forward_design_os.md` が商談再現・前進設計の中核OSであるのに対し、本OSは以下を補強する。

```txt
1次商談
↓
2次商談
↓
2次商談後フォロー
↓
2.5次商談 / 電話フォロー / 社内共有資料 / 決裁者接続
↓
3次商談 or 中長期ナーチャリング
```

重要なのは、2次商談後に「検討状況いかがでしょうか」で終わらせないこと。

2次商談後は、相手社内で案件が進む条件を分解し、次に必要な一手を選ぶフェーズである。

---

## 1. 2次商談後フォローOS

2次商談後のフォローでは、商談の感触ではなく、相手社内で何が起きているかを見る。

悪いフォロー：

```txt
ご検討状況いかがでしょうか
↓
検討中です
↓
失注・停滞
```

良いフォロー：

```txt
2次商談後の社内共有状況を確認する
↓
誰に共有されたかを確認する
↓
止まっている論点を分ける
↓
決裁者・関与者接続の余地を探る
↓
次回商談・電話フォロー・社内共有資料のどれが必要か判断する
```

### 1.1 標準確認項目

2次商談後は、以下を確認する。

```txt
社内共有は進んでいるか
誰に共有されたか
決裁者・関与者は誰か
何が懸念になっているか
費用対効果で止まっているのか
運用負荷で止まっているのか
対象範囲で止まっているのか
次回商談に誰を呼ぶ必要があるか
社内共有用資料が必要か
電話フォローで足りるか
2.5次商談が必要か
```

### 1.2 次アクション分類

2次商談後の打ち手は、必ず以下のどれかに分類する。

```txt
A. 3次商談化
決裁者・関与者を巻き込んで前に進める

B. 2.5次商談化
正式な3次商談の前に、温度感・社内共有状況・懸念を整理する

C. 電話フォロー
次回商談化する前に、軽く状況確認して停滞理由を拾う

D. 社内共有資料化
担当者が上申・共有しやすい判断材料を作る

E. 中長期ナーチャリング
今すぐではないが、検討時期・社内課題・次接点を管理する
```

2次商談後は「次回商談を取るか／取らないか」だけで判断しない。

相手社内で止まっている論点を特定し、次に必要な一手を選ぶ。

---

## 2. IS・商談フェーズ分離OS

ISと商談者を同じ役割として扱わない。

ISは架電段階の前工程。  
商談者は商談内の深掘り・温度感把握・次回設計を担う後工程。

```txt
IS
= 商談前の初期接点づくり・仮説収集・温度感の入口確認

商談者
= 商談内での課題整理・温度感把握・次回商談設計

PM / SM
= 全体戦略・勝ち筋設計・資料設計・決裁導線設計
```

### 2.1 ISで拾うべき情報

ISの目的は、単にアポを取ることだけではない。

商材によっては、以下の情報を拾うことで、商談設計の精度が上がる。

```txt
どの課題に反応したか
誰が関与しそうか
決裁者接続の余地があるか
どの切り口なら受付突破しやすいか
どの表現で拒否されたか
商談時に深掘りすべき論点は何か
```

ただし、ISと商談者の役割を混同してはいけない。

```txt
IS = 商談前の仮説づくり
商談者 = 商談中の課題整理
フォロー担当 = 商談後の前進条件整理
```

この3つを分けることで、営業活動を「点」ではなく「工程」として改善できる。

## 3. SP＋IS比較検証OS

通常架電で成果が出ない場合、すぐにトーク全体を否定しない。

架電結果は以下の3つに分けて検証する。

```txt
1. リストの問題
そもそも対象企業が合っていない

2. トークの問題
入口の切り口・表現・順番が合っていない

3. 架電者スキルの問題
受付突破、切り返し、深掘り、ログ粒度に差が出ている
```

特に商材理解やヒアリング力が必要な案件では、通常架電とは別に、重要先・差分検証用のISを一部差し込む選択肢を持つ。

目的はSPを置き換えることではない。

目的は、通常架電との差分を見て、勝ちパターンを抽出することである。

### 3.1 比較検証で見る項目

```txt
受付突破率
担当者接続率
課題感の取得率
決裁者・関与者情報の取得率
商談設計に使えるログの量
相手が反応した言葉
拒否された言葉
次回商談に使える仮説
```

### 3.2 実施ルール

```txt
最初から大きく広げない
重要リストに限定する
期間・件数に上限を設ける
通常架電と比較できる形で残す
アポ数だけで評価しない
商談設計に使える情報が取れたかで評価する
```

### 3.3 架電者条件定義

架電者を指名できない場合は、人名ではなく条件で定義する。

```txt
深掘りヒアリングができる
ログの粒度を高く残せる
相手の反応に合わせて質問を変えられる
アポ獲得だけでなく、商談設計に使える情報を残せる
```

勝ちパターンが見えたら、SP側のトーク・ヒアリング項目・ログ記載ルールに展開する。

## 4. ダッドウェイ様・ほしゆう様からの抽出

### 4.1 ダッドウェイ様：2次商談後の温度感確認・決裁者接続

ダッドウェイ様は、2次商談でサービス理解は一定進んでいるものの、次の論点は「導入するかどうか」ではなく、社内で誰が判断し、どのような形で検討を進めるかに移っている。

次フォローでは、以下を確認する。

```txt
2次商談後に社内共有が進んでいるか
誰に共有されているか
決裁者・関与者に接続できる余地があるか
導入検討にあたって懸念になっている点は何か
次回商談を設定する場合、誰の同席が必要か
```

いきなり3次商談にするより、2.5次商談または電話フォローで温度感と社内共有状況を確認する選択肢を持つ。

### 4.2 ほしゆう様：中長期検討・役員相談材料化

ほしゆう様は、役員相談が必要であり、導入検討には半年から1年程度のリードタイムが想定される。

短期即決型ではなく、中長期検討型の商談として扱う。

次フォローでは、以下を確認する。

```txt
役員への共有が進んでいるか
共有した際にどの論点が引っかかっているか
半年から1年の検討期間の中で、いつ何を整理すべきか
メンタル不調、退職、従業員コンディションに関する課題をどこまで社内課題として扱えるか
次回商談を行う場合、誰を巻き込むべきか
```

今すぐのクロージングよりも、社内検討の入口を作り、役員相談に必要な判断材料を整えることを優先する。

## 5. 定例資料への落とし込み

新規商談がない場合でも、定例資料は「商談件数が少ない」という報告で終わらせない。

以下の構成で、既存商談の前進設計としてまとめる。

```txt
1. 今月の商談件数・新規創出状況
2. 既存商談の2次商談後ステータス
3. 案件別に止まっている論点
4. 次フォローの目的
5. 2.5次商談 / 電話フォロー / 社内共有資料 / 決裁者接続の選択
6. 次回までに確認すべき事項
```

新規商談が出ていない現状は課題だが、既存商談の2次商談後フォローを強化することで、担当者止まりを防ぎ、社内検討・決裁接続に進みやすい状態を作る。

---

## 6. 一文定義

2次商談後に案件を放置せず、電話フォロー・2.5次商談・社内共有資料・決裁者接続に分解して、相手社内で案件が進む状態を設計するOS。
