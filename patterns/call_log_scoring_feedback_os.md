# 共通OS｜架電ログ再学習・リスト評価改善モデル

## 1. 基本思想
営業電話を単なるアポ獲得作業ではなく、**リスト評価AI・市場仮説・トーク精度を鍛える実験装置**として扱う。

通常の営業支援は、

> リスト作成 → 架電 → アポ数を見る

で止まりやすい。

このOSでは、

> リスト評価 → 上位群へ集中架電 → 反応ログ取得 → スコアリング基盤再学習 → 勝ち筋強化

までを1サイクルとして設計する。

## 2. 運用フロー
1. 企業情報からリスト評価を行う
2. 評価上位企業を優先して架電する
3. 担当者接続後は、表面的なニーズ確認ではなく深掘りヒアリングを行う
4. 架電ログから実際の反応を取得する
5. 反応内容をもとにスクリプトを改善する
6. アポ化・非アポ化の理由をもとに評価指標を再調整する
7. 次回リスト生成・次回架電台本に反映する

## 3. 架電担当者の位置付け
架電担当者は、単なるアポ獲得者ではない。

**市場仮説を検証する一次リサーチャー**として扱う。

見るべきものは、アポ化だけではなく、相手の反応・断り文句・刺さった質問・想定課題とのズレである。

## 4. 架電ログで見るべき項目
- 高スコア企業の反応は本当に良かったか
- 受付拒否が多い業界・部署はどこか
- 担当者接続後に刺さるワードは何か
- 断り文句は何か
- 想定課題と実課題がズレていないか
- どの企業属性がアポ化しやすいか
- どの質問で相手が話し始めるか
- どのトークで情報が閉じるか
- どの切り口なら「今はないです」で終わらず、現状説明が始まるか

## 5. KPIの扱い
架電KPIは成果管理だけでなく、仮説検証指標として見る。

見るべき代表KPI：
- 架電数
- 着電率
- 受付突破率
- 対象者接続率
- 担当者会話率
- アポ率
- 高評価リスト別アポ率
- 高評価リスト別拒否率
- トーク切り口別の会話継続率
- 断り理由別の比率
- 完全新規アポ率
- 過去接点フォローアポ率
- 担当者条件取得率
- 具体テーマ反応率

重要なのは、件数ではなく**仮説検証密度**。

## 6. リスト評価AIへの戻し方
架電結果は、次の分類でリスト評価指標に戻す。

| 観点 | 戻し方 |
|---|---|
| 高スコア企業が好反応 | 評価指標を維持・強化 |
| 高スコア企業が低反応 | 評価条件の過大評価を疑う |
| 低スコア企業が好反応 | 新しい勝ちシグナルとして抽出 |
| 受付拒否が集中 | 業界・部署・代表番号ルートの問題として分離 |
| 担当者接続後に失速 | スクリプト・課題導入・質問設計の問題として分離 |
| アポ化せず情報が取れた | 市場仮説の教師データとして保存 |
| 過去接点ではアポ化するが完全新規で失速 | 過去文脈依存と判断し、完全新規用の課題導入を別設計する |
| 「何の話か分からない」と拒否 | 専門ワードの抽象度が高すぎるため、相手の業務場面に変換する |

## 7. 汎用トーク設計原則
弱い聞き方：
- 今困っていますか？
- 外注していますか？
- 導入予定はありますか？
- 20分ほどサービス説明させてください

強い聞き方：
- 最近、〇〇と△△を同時に求められるケースが増えていると思うのですが、御社ではこの優先順位はどこで整理されていますか？
- 案件ごとに判断が変わる部分かと思うのですが、社内で完結されていますか？それとも外部も含めて調整されていますか？
- いま顕在化している課題というより、判断が難しくなる場面があるかだけ確認させてください。
- 今すぐ外注先を変える話ではなく、今後こうしたテーマが出た際の比較材料として整理できればと思っています。

売りたい機能を説明する前に、相手の業務上のトレードオフ・優先順位・制約条件を質問で顕在化する。

## 8. T2案件への適用例
T2案件では、以下の制約同時発生を課題導入に使う。

- 高機能化
- AI対応
- 小型化
- 省電力化
- 回路・基板・組込み・筐体・ソフトウェアの複合調整

代表トーク：

> 最近、高機能化・AI対応・小型化・省電力化を同時に求められる案件が増えていて、設計段階でどこを優先するかの整理が難しくなっている企業様が増えています。  
> 御社では、このあたりの優先順位は設計段階で明確に決まっていますか？  
> それとも案件ごとに調整されることが多いですか？

## 9. 完全新規アポ創出型

### 9.1 背景
過去接点フォローでアポが取れても、完全新規で同じトークが通用するとは限らない。

完全新規では、相手に前提文脈がないため、専門ワードを先に出すと「何の話か分からない」「何を売りたいのか分からない」で会話が閉じる。

この場合、営業説明を強くするのではなく、相手が自社の業務場面として想像できる入口へ変換する。

### 9.2 完全新規の設計原則
1. サービス説明ではなく、相手の業務上の判断場面から入る
2. 抽象ワードを、相手の現場で起きる具体的な場面に言い換える
3. アポ打診前に、相手が答えやすい確認質問を1つ挟む
4. 「売り込み」ではなく「該当可能性の整理」として商談化する
5. アポ数だけでなく、会話継続・具体テーマ反応・担当者条件取得をKPI化する

### 9.3 T2型の完全新規トーク
受付突破：

> 製品開発・装置開発で、基板設計や組込みソフト、AI処理まわりの外部相談を見られている技術責任者の方にご案内です。  
> 新規の大きな外注依頼というより、既存の開発体制だけでは切り出しにくい設計領域について、情報提供の件でご連絡しております。

本人接続後：

> 最近、PCで処理していた画像処理・AI判定・制御処理を、装置側や端末側に寄せたいという相談が増えております。  
> その際に、処理性能・消費電力・発熱・サイズ・コストのどれを優先するかで設計が詰まりやすいのですが、御社ではこのあたりは現在どのように整理されていますでしょうか？

アポ打診：

> 今すぐ外注先を変えるという話ではなく、今後こうした設計テーマが出た際の比較材料として、事例ベースで一度整理できればと思っております。  
> 20分ほどオンラインで、御社に該当しそうな領域があるかだけ確認させていただけないでしょうか。

### 9.4 汎用化できる型
T2に限らず、専門サービスの完全新規営業では以下の変換を行う。

| 売り手側の言葉 | 買い手側の業務場面 |
|---|---|
| AI対応 | 既存業務・装置・システムにAI処理を組み込む場面 |
| DX支援 | 現場で二重入力・属人判断・手戻りが発生する場面 |
| 設計支援 | 社内リソースだけでは優先順位整理が難しい場面 |
| コンサルティング | 仕様・体制・投資判断を決める前に論点を整理する場面 |
| 外注先 | 既存体制で切り出しにくい部分の比較材料 |

### 9.5 検証KPI
- 完全新規アポ率
- 完全新規の対象者通話後アポ率
- 受付で担当者条件を取得できた率
- 「何の話か分からない」拒否率
- 具体テーマ反応率
- アポ未満の情報取得率
- 過去接点フォローとの差分

## 10. 結論
このOSの本質は、

> リスト評価AI × 深掘り架電 × ログ再学習

である。

架電は市場デバッグ。
ログは報告書ではなく、次のリスト・次の台本・次の勝ち筋を強くする教師データとして扱う。

完全新規アポ創出では、専門ワードを売るのではなく、相手の業務場面に変換された「判断が難しくなる瞬間」を提示する。
