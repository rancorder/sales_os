# SLIDE_OS_POLICY｜営業OS スライド開発規約

## 0. 目的

この規約は、GitHub → Vercel で公開するHTMLスライドを、営業OSの正本運用に統合するためのルールである。

営業資料を単発制作物ではなく、商談分析・勝ち筋・判断履歴・反応学習を含む「営業資産」として扱う。

---

## 1. 基本思想

```txt
商談ログを読む
↓
勝ち筋を抽出する
↓
スライド構成に変換する
↓
HTMLスライドとして実装する
↓
VercelでURL確認する
↓
商談後の反応を学習させる
```

この循環を回すことで、スライドの品質と制作速度を同時に上げる。

---

## 2. 正本ルール

### Driveに置くもの

```txt
商談動画
商談文字起こし
顧客資料
過去PowerPoint
過去PDF
画像・動画などの重い素材
機密情報を含む原本
```

### GitHubに置くもの

```txt
slide_spec.md
storyboard.md
talk_script.md
decision_log.md
review_checklist.md
HTML/CSS/JS
Vercel URL
スライドパターン
```

### GitHubに置かないもの

```txt
顧客の生ログ
電話番号
メールアドレス
個人情報
契約書
顧客内部資料の原本
未加工の商談動画
```

---

## 3. 必須ファイル

各スライド案件は、原則として以下を持つ。

```txt
slides/projects/{client}_{theme}_{date}/
├── slide_spec.md
├── storyboard.md
├── talk_script.md
├── decision_log.md
├── review_checklist.md
├── index.html
├── style.css
├── main.js
└── vercel_url.md
```

HTMLが1ファイル完結の場合でも、`slide_spec.md` と `decision_log.md` は必ず残す。

---

## 4. スライド設計の順番

### 悪い順番

```txt
デザインから作る
サービス説明から作る
過去資料をそれっぽく流用する
```

### 良い順番

```txt
相手の状況
↓
前回商談の論点
↓
見えている課題
↓
まだ見えていない課題
↓
提案の位置付け
↓
判断材料
↓
次回アクション
```

営業OSでは、必ず **相手の頭の整理 → 提案接続** の順番にする。

---

## 5. スライド種別

### 5.1 商談プレゼン型

外向き商談資料。

```txt
表紙
前回整理
課題整理
見えていない論点
提案
運用イメージ
費用/進め方
次回アクション
```

### 5.2 役員判断型

決裁者向けの短い判断資料。

```txt
なぜ今か
放置リスク
導入効果
費用対効果
実施範囲
判断ポイント
```

### 5.3 商談FB型

PM・社内共有向け。

```txt
商談結果
温度感
仮説との一致/ズレ
次回化理由
懸念
次の一手
```

### 5.4 デモ導入型

操作画面やプロトタイプを見る前の橋渡し。

```txt
前提課題
デモで見るポイント
画面イメージ
操作の流れ
導入後の変化
```

### 5.5 インフォグラフィック型

複雑な仕組みを一枚で理解させる資料。

```txt
全体像
流れ
役割分担
学習循環
実例
一言まとめ
```

## 6. 品質基準

### 商談品質

- この資料で相手に何を判断してほしいか明確か
- 次回アクションにつながるか
- 商談ログから抽出した勝ち筋が入っているか
- 汎用サービス説明になっていないか

### 構成品質

- 1枚目から順番に納得感があるか
- サービス説明が早すぎないか
- 相手の課題整理が先にあるか
- 1スライド1メッセージになっているか

### 表現品質

- 難しい言葉を使いすぎていないか
- 相手の社内共有に使える言い方か
- 決裁者が読んでも意味が分かるか
- 誇張や断定が強すぎないか

### 実装品質

- Vercelで表示崩れがないか
- スマホ/PCで読めるか
- 画面幅を変えても破綻しないか
- URL共有できる状態か

### 資産化品質

- decision_logに判断理由を残したか
- 案件カルテにスライド履歴を追記したか
- 再利用できる型をpatternsへ抽出したか

---

## 7. Vercel運用

- GitHub連携で自動デプロイする。
- 本番URLとPreview URLを区別する。
- 商談前確認はPreview URLでもよい。
- 顧客提示・商談本番で使う場合は本番URLを `vercel_url.md` に記録する。
- URLを案件カルテ7章にも追記する。

---

## 8. 案件カルテへの追記形式

各 `clients/{client}.md` には必要に応じて以下を追記する。

```md
## 7. スライド開発履歴

| 日付 | 用途 | スライド種別 | GitHub path | Vercel URL | Drive ID | 判断ログ |
|---|---|---|---|---|---|---|
| YYYY-MM-DD | {用途} | {種別} | slides/projects/{project} | {URL} | {Drive ID} | decision_log.md |
```

---

## 9. 禁止事項

- HTMLだけ作って判断理由を残さない。
- 商談ログにない事実を足す。
- 顧客機密や個人情報をGitHubへ直置きする。
- 見た目だけ整えて商談上の勝ち筋を薄める。
- PowerPointの代替としてだけ扱う。

---

## 10. この規約の一言要約

```txt
スライドは成果物ではなく、営業判断を実装したプロダクトである。
```
