## 0. 一言定義

この案件は、Microsoft 365 / Office 365 セキュリティ支援を、過去資料請求・Webinar・M365導入予定企業などの温度感あるリードから商談化し、SU担当者をフォローコール担当から商談者へ育成する案件である。

単なる架電代行ではなく、以下を含む複合運用である。

```txt
SP架電
↓
資料請求・過去接点フォロー
↓
SUフォローコール
↓
商談化
↓
商談同席
↓
商談者育成
↓
商談主導
↓
商談後登録・連絡・録画・Mazrica運用
```

## 1. 抽出された勝ち筋

### 1.1 M365導入・移行タイミングを拾う

反応が強いのは「セキュリティで困っている」と明言する企業だけではない。

強いシグナル：

```txt
Microsoft 365 / Office 365 をこれから導入する
Microsoft 365 を全社展開しようとしている
新しく入れるサービスのセキュリティが不安
予算時期が近い
検討中心人物と接続している
```

このシグナルがあるリードは、資料請求リードの中でも優先度を上げる。

### 1.2 セキュリティ担当だけで止めない

セキュリティ商材では、最初の接点が情シス・セキュリティ部門でも、実際に話すべき部署が経営企画・企画部・DX推進側にある場合がある。

営業OS上のルール：

```txt
商談後に必ず確認すること：
・本当に検討している部署はどこか
・予算や導入判断を持つ部署はどこか
・次に話すべき人は誰か
・情シス/セキュリティ部門だけで判断できるのか
```

### 1.3 SCS評価制度を経営入口にする

「M365使っていますか」だけでは、相手に製品営業として受け取られやすい。  
SCS評価制度やサプライチェーン上のセキュリティ要件を入口にすると、経営・取引条件の話に上げやすい。

使う文脈：

```txt
サプライチェーン全体でセキュリティ水準の確認が進む
取引先から求められる対策が増える
中小企業でも最低限の対策が必要になる
M365セキュリティ / SOC / 監視運用と接続できる
```

### 1.4 過去接点リードにSUを当てる

membersチャット日報ベースでは、過去資料請求系リストでSUが高い対象者通話率・アポ率を出している。

抽出KPI：

| 指標 | 合計 |
|---|---:|
| 架電数 | 79 |
| 受付拒否 | 0 |
| 対象者通話 | 33 |
| 本人資料送付 | 6 |
| アポ取得 | 6 |
| 対象者通話率 | 約41.8% |
| アポ率 | 約7.6% |

判断：

```txt
SUは完全新規より、過去資料請求・資料送付・Webinar・体験会など、温度感のあるリードに当てると効果が出やすい。
```

## 2. 主要課題

### 2.1 商談後タスク漏れ

サンクスメール、リマインド、SC登録、Mazrica登録、Teams報告、録画確認が属人管理になり、漏れが発生している。

原因：

```txt
・作業が複数ツールにまたがる
・誰がどこまでやるかが曖昧
・送信予約、下書き、登録完了の確認が弱い
・商談後タスクがチェックリスト化されていない
```

対策：

```txt
商談終了後チェックリストを作る。
完了条件を「やったつもり」ではなく「登録/送信/予約/報告済み」で確認する。
```

### 2.2 Outlook / Teams / SC / Mazrica 運用が複雑

この案件では通常のアポ獲得後処理では足りない。

必要工程：

```txt
Outlookで予定確認
Teams URL発行
Teamsチャットで報告
SalesCrowd登録
Mazrica登録
クライアントメールで日程確定・サンクス・リマインド
録画格納
商談結果共有
```

対策：

```txt
案件別SU運用マニュアルを作成し、架電前・アポ取得時・商談前・商談後で分ける。
```

### 2.3 商材理解不足

M365セキュリティは専門性が高く、台本を読むだけでは商談主導が難しい。

不足しやすい知識：

```txt
M365 / Office365
E3 / BP
EPP / EDR
FW / NW
SOC / MSOC / MDR
VPN / DR
AD / MDM / 資産管理
SCS評価制度
Microsoftパートナー文脈
```

対策：

```txt
商材理解シート、専門用語シート、代表商談の質問集を作る。
```

### 2.4 リスト精査不足

発生している問題：

```txt
・対象外属性の混入
・同一先への重複架電
・Google利用企業への訴求ズレ
・M365利用状況不明企業の混在
・資料請求先の温度感差が未整理
```

対策：

```txt
S/A/B/C/D判定を導入する。
M365利用・導入予定・部署・予算時期・対象外条件で振り分ける。
```

### 2.5 架電者品質差

台本読みの架電者が受付拒否率を悪化させるケースがある。

見るべき指標：

```txt
受付拒否率
その他拒否率
対象者通話率
台本棒読み率
ログ粒度
反応に応じた質問変更
商談設計に使える情報取得
```

### 2.6 録画・音声取得不備

録画不可、録画できても音声なし、Teams音声不良などが発生。  
商談分析OSにとって、録画・音声欠落は致命的。

対策：

```txt
商談前に録画方法・音声入力・録画開始を確認する。
商談後に録画ファイルの音声有無を確認する。
音声がない場合は当日中に商談メモで補完する。
```

### 2.7 アラート対応基準がない

SU稼働起因で解約要望を含むアラートが発生。  
上長説明・改善提案が必要な状態となった。

対策：

```txt
アラート対応OSを作る。
事実整理、影響範囲、原因、暫定対応、再発防止、期限、担当者、顧客説明を1枚で整理する。
```

## 4. 次の改善アクション

### 即時

1. 商談後チェックリストを作成し、全商談で運用する。
2. SU担当者向けのOutlook / Teams / SC / Mazricaマニュアルを作る。
3. M365セキュリティ商材理解シートを作る。
4. リード判定S/A/B/C/Dを定例資料へ入れる。
5. アラート説明用の改善提案書を作る。

### 中期

1. 代表商談・先方商談ログから質問集を作る。
2. SU担当者の商談同席ログを台本に反映する。
3. 3月下旬を目安に、先方同席ありのSU主導商談へ移行する。
4. 独り立ち後も商談後チェックと録画確認は必須にする。

---

## 5. 重要な結論

```txt
この案件の本質は、アポ数を増やすことではない。

M365セキュリティという専門性の高い商材を、
温度感あるリードから商談化し、
SU担当者がフォローコール・商談・商談後処理まで担える状態にすること。

そのためには、リスト精査、商材理解、商談同席、商談後チェック、
Outlook/Teams/Mazrica運用、アラート対応までを一つの工程としてOS化する必要がある。
```
