# _research_protocol.md｜営業OS 新規立ち上げプロトコル（運用前）

資料しかない＝まだ架電/商談が回っていない企業を、ゼロから立ち上げる手順書。
**人（チャットのClaude）もエージェント（Codex/Claude Code）も、同じこれを読んで動く。**

運用後（ログがある企業）の改善は `_analysis_protocol.md` を使う。これは**運用前専用**。

---

## 0. 全体の流れ

```
① 事前リサーチ（仮説づくり）
        ↓
② 戦略たたき（勝ち筋の決定）
        ↓
③ 架電台本 → 商談台本
        ↓
④ 商談スライド
```

**順番は固定。**戦略を固める前に台本・スライドを作らない。
リサーチが浅いと全部が浅くなるため、①に最も時間をかける。

---

## 1. 事前リサーチ（運用後と目的が違う）

運用後リサーチ＝「どこで落ちているか」を当てる。
**運用前リサーチ＝「どこに刺せば当たるか」を仮説する。** 改善材料がないので、仮説の質が全てを決める。

### 必ず埋める6点（順番に意味がある）

1. **商材の核（USP）**
   - 一言で何屋か／他社と何が違うか。"刺さる一言"を探す。
   - 例：アスト「設計を増やすでなく社内設計者の負荷を逃がす」。

2. **ターゲットの歪み（課題仮説）**
   - 狙う業界の構造的な痛み。市況・法改正・コスト構造の外部フック。
   - 例：参鋼「ロットの壁の下」／旭園「特定技能の新規受入停止」。

3. **誰の・どの予算で買うか**
   - 担当部署と予算科目。ここを外すと刺さらない。
   - 例：フードサプライズ「食費でなく販促費・成約御礼費」。

4. **想定される拒否（先回り）**
   - 商材から「どう断られるか」を予測し、切り返しを仕込む。
   - 例：花雪「AI不信」「AI開発が大きすぎて自分ごと化できない」。

5. **競合と代替手段**
   - 相手は今その課題を何で解決しているか（内製/既存外注/他社/我慢）。
   - 例：WSE「既存外注の置き換えでなく隙間に入る」。

6. **勝ち筋の仮説**
   - 1〜5から「どの相手の・どの痛みに・どう言えば二次商談に乗るか」を1〜2行。
   - これが②戦略たたきの背骨になる。

### 材料の取り出し元（運用前でも揃う）
- **アイドマの取材項目表 / インタビュー項目**（各社フォルダにある）→ 1,3,5
- **営業資料・会社案内 pptx/pdf**（6章のfileId）→ 1,2
- **HP・業界ニュース**（外部検索）→ 2,4の市況フック
- **似た既存カルテ**（製造業設計ならアスト/WSE/参鋼、映像ならKANSHA/HITO Film）→ 過去の勝ち筋を流用 ★OS化の旨み

### リサーチ出力フォーマット（カルテ2章に格納）
```
## 2. 商材の核 / USP（事前リサーチ）
- USP：{刺さる一言}
- ターゲットの歪み：{外部フック}
- 予算の出どころ：{部署・科目}
- 想定拒否と先回り：{拒否→切り返し}
- 競合・代替：{今の解決手段}
- 勝ち筋仮説：{誰の・どの痛みに・どう言うか}
```

## 2. 戦略たたき（勝ち筋の決定）

リサーチ6点を踏まえ、**一次接点〜二次トスアップまでの設計図**を1枚で決める。

### 出力フォーマット（`戦略たたき_{社名}.md`）
```
# 戦略たたき｜{社名}
## ゴール定義
- 一次接点のゴール：{何を取れたら勝ちか＝原則「二次商談トスアップ」}
## ターゲット設計
- 第1優先：{業種・規模・状態} → 理由
- 第2/第3優先：…
## メッセージ設計
- 冒頭フック（刺さる一言）：
- 自分ごと化させる問い：
- 予算ブロックの外し方：
## 想定反論マップ
- 反論①→切り返し / 反論②→切り返し …
## 勝ち筋（1〜2行）
```

戦略が固まったら、INDEXのステータスを「戦略策定」に更新。

## 3. 台本（架電 → 商談）

戦略の「メッセージ設計／反論マップ」を、現場の発話に落とす。

### 3-A. 架電台本（一次接点・受付突破〜本人接続〜アポ）
- 冒頭1文に商材説明を入れない（営業判定を避ける）。戦略の冒頭フックを使う。
- 状況別分岐（常時/必要時/対象外 など）で対象外を早く切る。
- ゴール＝二次商談化。**「説明する」でなく「課題を切り分ける」**設計。
- 禁止表現は戦略の「想定拒否」から逆算して列挙。

### 3-B. 商談台本（一次商談・ヒアリング→二次トスアップ）
- 一次商談はヒアリング主体。最後に二次商談へトスアップ。
- フロー：現状確認→課題特定→ズレ言語化→価値提示→BANT→クロージング。
- 反論マップを切り返しとして組み込む。

> 既存の良い台本例：花雪 推奨架電台本（"切り分ける営業"）、ビスタ 商談フローチャート、アスト 対応範囲3分類。

## 4. 商談スライド

台本の流れをそのまま画面に。**1スライド＝1メッセージ**、各ページに「思わせること／想定反論／反論が消える理由」を裏に持たせる（kotonoha協賛スライドv3の設計思想）。

### 標準構成（9枚目安）
1. 表紙（刺さる一言をコピーに）
2. 現状確認（相手に課題を選ばせる）
3. 課題の診断（どこが原因かを分解）
4. 再定義（"当てにいく状態"等、課題を言い換える決定打）
5. サービス概要（やることをシンプルに）
6. 実画面/実例（実在感）
7. 価値・差別化（応募増でなく再現性、等）
8. メリット3点＋実績
9. ネクストステップ（二次商談の日程打診）

### 出力
- 制作はpptxgenjs or 16:9インタラクティブHTML（ひでさんの標準）。
- 視覚QA：LibreOffice headless→pdftoppm、またはPlaywright/Chromium。
- 書き言葉は「貴社」。

## 6. 運用前→運用後の引き継ぎ
- 架電が始まったら、このプロトコルの成果物（戦略・台本・スライド）はそのまま残し、
  以降の改善は `_analysis_protocol.md`（4章に分析済みマーカーで追記）へ移行する。
- つまり「立ち上げ＝research_protocol」「改善＝analysis_protocol」で2本立て運用。

---

## 7. 注意
- 運用前は実データがない。リサーチは**仮説**であることを明記し、断定しない。
- HPや業界ニュースを使うときは出典を残す。
- 動画・実績の数値は、確認できないものを作らない（{未確認}）。
